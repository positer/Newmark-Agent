# dev-0.0.10 Auto routing industrial strategy record

- Timestamp: 2026-07-15 +08:00 (Asia/Shanghai)
- Scope: engineering strategies adopted for Newmark's local global/provider-scoped Auto router
- Product boundary: one resolved model answer per user turn; no parallel answer fusion and no external black-box prompt router
- Product-intent input: the user-provided DeepSeek discussion, treated as design context rather than an authoritative technical specification

## Deterministic selection order

1. Build the enabled candidate pool under global or stable provider-ID scope and the optional explicit subset.
2. Reject preview, context, modality, tool, JSON, privacy, budget, region, protocol-parameter, validation, and open-circuit mismatches.
3. Classify the task locally into chat, coding, reasoning, long-context, vision, image generation, tool use, or ComputerUse without a hot-path LLM call.
4. Estimate task-domain quality with a conservative Beta prior: `q=(successes+2)/(attempts+4)`. Transport failure does not count as task-quality evidence.
5. Keep only candidates inside the selected quality-loss band.
6. Rank the band using declared cost, observed latency/throughput/reliability/tool validity, cache affinity, and bounded preference. Unknown price is worst-case; unknown health uses a neutral prior.
7. Keep the five-minute incumbent unless it violates a hard constraint or a challenger exceeds the configured utility threshold.

## Safety and audit boundary

- Same model names are not equivalent identities. Cross-endpoint switching requires an explicit logical model group.
- Provider URL, credential, protocol, and adapter are re-resolved for every attempt from the chosen deployment.
- Content-policy refusal, invalid schema, unknown tool, and authentication failure do not trigger a cross-model bypass.
- A visible first token prevents transparent switching. Side-effect tools prevent replay; read-only results may be recovered only from persisted structured results.
- Route audit contains IDs, policy/catalog versions, task classes, exclusions, normalized components, attempt timing/status/error class, and final status. It excludes credentials, prompt text, tool parameters, file content, and complete provider responses.
- Learned preference cannot cross a hard constraint or quality band, requires three same-domain events in thirty days, decays over time, and has a clear/reset path.

## Explicitly rejected for dev-0.0.10

- online random exploration or Thompson sampling;
- a second LLM classifier on the request hot path;
- parallel answer generation/fusion;
- semantic response caching;
- moderation/refusal bypass by silently switching models;
- cross-scope fallback or forced calls while all candidates are circuit-open;
- automatic admission of unvalidated catalog additions;
- embedding LiteLLM, Portkey, or another gateway/runtime dependency.
