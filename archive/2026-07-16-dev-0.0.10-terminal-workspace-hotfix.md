# dev-0.0.10 Terminal And Legacy Workspace Hotfix

Date: 2026-07-16 (Asia/Shanghai)

## Reported Regressions

1. Collapsing the bottom terminal removed the entire terminal surface, including the only reopen control.
2. Selecting an older internal workspace failed while the configured Agent backend was WSL.

## Root Causes

- `#bottom` used `height: 0` in the collapsed state while `#bottom-header` and `#bottom-toggle-btn` were children of that same element. The control therefore became unreachable after collapse.
- Windows main and WSL Agent processes shared `~/.Newmark/Work/Local.json`, `External.json`, and `State.json`. The WSL host itself and each `ConversationKernel` runner constructed a normal `WorkspaceManager`, so both operating systems repeatedly normalized and saved the same paths. Windows paths became `/mnt/c/...`; subsequent cross-host `path.resolve` calls produced repeated mixed paths such as `/mnt/c/.../C:\\mnt\\c.../C:\\Users...`.

## Correction

- The terminal collapsed height is 34 px and its header has a 33 px minimum height. Only the body collapses; the toggle remains visible.
- `normalizeHostWorkspacePath` recognizes Windows drive paths, WSL `/mnt/<drive>` aliases, and recovers the final valid Windows absolute path from previously repeated mixed-host strings.
- `WorkspaceManager` supports a detached mode that neither scans nor saves the shared workspace registry.
- The WSL host and every conversation runner are created through one detached `createWslAgent` factory. Windows main remains the sole owner of the workspace registry.
- The real registry was backed up under `~/.Newmark/Backups/dev-0.0.10-workspace-registry-20260716-132904` and `~/.Newmark/Backups/dev-0.0.10-workspace-registry-20260716-133221`, then migrated to stable Windows paths.
