## Glass contract

The General settings surface keeps one slider. Its value `A` is opacity in percent and its complementary transparency is `T = 100 - A`. Glass width is now inverted as requested:

- `B = 20 * T / 100`
- level 1 blur: `0.4B`
- level 2 blur: `0.8B`
- level 3 blur: `B`
- level 1/2/3 alphas: `0.75A`, `0.80A`, and `0.85A`, with `A` normalized to `0..1`

Executable anchors cover opacity `0/25/50/85/100`, including the exact blur triples `8/16/20`, `6/12/15`, `4/8/10`, `1.2/2.4/3`, and `0/0/0` pixels. The description now exposes opacity, complementary transparency, total glass width, and all three blur levels.

`input` previews CSS variables only. `change` persists the explicit `glassAlpha` field, and both Electron IPC and HTTP config paths map it to `ui.glass_alpha`. Backend and renderer hydration use nullish fallback semantics, so a legitimate `0` remains fully transparent instead of becoming the `0.85` default. Theme RGB channels remain in CSS; the slider changes only alpha and blur variables, so a light theme is not overwritten with dark RGBA values. Every `backdrop-filter` glass surface now references one of the three tunable blur variables.

## Startup and Browser lifetime

The blocking startup path now contains configuration/Agent initialization, core services, the current conversation runtime snapshot, preload-backed renderer state, and two rendered animation frames. File tree, right status, Flows, terminal, automation, WSL discovery, sidecar startup, and update discovery run after promotion through cancellable deferred schedulers. Deferred failures are warnings and cannot reject the first-frame path.

The Browser panel contains no source `<webview>` node. The first Browser-tab activation or Browser-tool request creates one guest through a single-flight promise, and creation is held until at least five seconds after renderer start. The guest always uses `persist:newmark-browser`. Leaving Browser schedules a 60-second idle teardown; the renderer retains the last URL and recreates one guest on the next demand. Browser-tool activity reuses the existing preload event to refresh the idle window, and main-process registration wait was increased to 12 seconds so a cold request can safely cross the five-second creation floor.

## Conservative renderer reductions

- `build-ui-icons.cjs` now intersects source string references with the installed Lucide catalogue and embeds only reachable symbols.
- Conversation and workspace list nodes append in 24-row animation-frame batches with generation cancellation.
- Prompt button refresh and editor render/completion work coalesce repeated input events into one animation frame.
- File tree, right status, Flows, and terminal hydration are staggered through idle callbacks after UI promotion.

## 2026-07-16 static integration update

The measurements and 48-assertion count above remain the historical result of the original focused slice. The integrated startup contract has since changed in three material ways:

- Startup creates one visible `BrowserWindow` and directly loads final `index.html?startupPrewarm=1`. A fixed cover inside that page remains visible until the same `webContents` receives the accepted `state/rendered` acknowledgement; no separate prewarm popup or hidden candidate window exists.
- The successful first-run hot path does not navigate through a splash document. Static `startup.html` is copied into `dist/ui` and is loaded only in the same window after a required failure; retry reuses that window for the final page.
- The critical barrier validates a local/persisted conversation snapshot and performs a task yield plus layout read. Flow, terminal, file tree, and right status have no background hydration queue and start only on explicit demand. Warning-only main-process work is delayed by 0.5 seconds for automation, 12 seconds for WSL discovery, 15 seconds for update discovery, 30 seconds for conversation-runtime prewarm, and 60 seconds for the non-interactive sidecar.

The current UI build embeds 72 Lucide symbols. `startupPrewarmVerify` now passes 73 assertions, including first-attempt readiness-waiter recovery without a duplicate navigation and the separated startup-before-Browser/on-demand-Browser memory-budget source contracts, while the original five glass anchors and valid-zero persistence remain unchanged. Browser navigation requested before guest `dom-ready` is retained and replayed exactly once after attachment. The formal benchmark described below has now completed; fresh ZIP/MSI hashes remain intentionally unrecorded until their artifact-specific gates finish.

Interactive P50/P95/max, Browser-open P95, and four-times-CPU input P95 all remained within their limits. Startup private-byte P95 was below the 525 MiB absolute limit and 28.34% below the 696 MiB baseline. Browser-on-demand total and delta P95 remained below 696 MiB and 300 MiB respectively. Every first-five-second monitor observed zero Browser guests in all 20 runs; the later explicit demand created exactly one guest per run. This closes the timing, low-performance input, startup-memory, and Browser-memory benchmark gate without making any ZIP/MSI or publication claim.
