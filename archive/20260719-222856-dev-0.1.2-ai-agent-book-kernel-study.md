## Focused source decomposition

### Stable context and KV/Prompt Cache

Upstream evidence:

- `chapter2/kv-cache/agent.py` keeps the correct-mode message list alive and
  appends subsequent assistant/tool messages; comparison modes rebuild the
  context, shuffle tools, change system data, or flatten message structure.
- `chapter2/kv-cache/README.md` identifies stable system prompts, stable tool
  order and preserved structured history as cache-friendly invariants.

Newmark comparison:

- `Agent.buildSystemPrompt()` is cached and regression-tested for stable output.
- The native runner builds a deterministic tool surface and keeps one Kernel
  state across provider/tool subturns.
- Request-specific task focus is appended at request construction rather than
  persisted as user-authored system policy.

Gap: Newmark protects several cache invariants, but does not expose a unified
measurement of provider cached-input tokens, stable-prefix fingerprints, or a
regression gate proving that an internal refactor did not invalidate the prompt
prefix. This is an observability gap, not evidence that the current prompt is
incorrect.

### Compression and continuation

Upstream evidence:

- `chapter2/context-compression` compares no compression, independent summaries,
  combined summaries, context-aware summaries, citation-preserving summaries,
  and an 80%-threshold windowed policy.
- The strongest general principle is task-aware compression with explicit
  evaluation of success, token use, compression ratio, overflow and tool calls.

Newmark comparison:

- `transformContext()` requests compression and can return separate immediate
  request messages and durable replacement messages.
- The detailed continuation anchor is request-only; durable Kernel history is
  atomically replaced with the compacted set.
- Compression-side cancellation resumes the same Build and `runId`, with a
  bounded retry count and one final-result boundary.

Gap: the continuity mechanism is stronger than the upstream teaching demo, but
Newmark lacks a compact, reusable benchmark that scores fact retention,
unfinished-task retention, stale-task rejection, compression ratio and repeated
compression avoidance together.

### Tool disclosure and selection

Upstream evidence:

- Chapter 4 presents active selection as an execution-time capability request.
- `chapter8/active-tool-discovery` compares all-tools injection, one-shot
  retrieval prefiltering, and repeated `discover_tools` calls.

Newmark comparison:

- `ToolProvisionSession` exposes a compact catalog plus a bounded broker, grants
  exact schemas, preserves policy filtering, and refreshes the surface when
  provider/runtime identity changes.
- Broker traffic stays internal and does not pollute public conversation history.

Gap: functional provisioning coverage exists, including reachability of the
tool catalog, but there is no acceptance matrix joining injected-schema tokens,
provision round trips, task success and false-negative tool discovery on the
same workload.

### Parallel tools, interruption and queues

Upstream evidence:

- `chapter4/async-agent` routes all events through an inbox, classifies urgency,
  runs independent I/O concurrently, turns task completion into a new event,
  and preserves checkpointed task progress across sessions.

Newmark comparison:

- Provider tool batches use parallel execution and an all-receipts barrier.
- Guide uses the live Kernel steering queue and has accepted/applied/deferred/
  rejected receipts; late Guides are durably moved to the next continuation.
- Next is conversation-targeted and drains only after synchronous ownership of
  the following Build.
- Runtime supervision distinguishes normal completion, interruption, force stop
  and late worker errors.

Gap: Newmark has stronger production ownership and receipt semantics, but its
tests are distributed across multiple suites. A single response-trajectory
contract should report queue latency, interruption point, cancelled work,
recovery result, receipt completeness and final-answer count.

### State hints, stages and completion

Upstream evidence:

- `chapter5/coding-agent/system_state.py` injects tool-call count and TODO state.
- `chapter10/staged-system-prompt` assigns separate roles, prompts and tool sets
  to requirements, implementation and review, with explicit transition tools.
- The simple upstream loops commonly treat an assistant response without tool
  calls as completion.

Newmark comparison:

- Build/Plan/Goal and persistent plan state already provide explicit task state.
- `buildRequestTaskFocus()` keeps the latest real user message authoritative and
  treats Goal/unfinished Plan only as continuity anchors.
- `shouldStopAfterTurn()` additionally respects pending Memory Lab rebuilds,
  Guide/queue ownership and tool results.
- Work runs persist exactly one final result for normal completion; failed and
  interrupted runs do not synthesize success.

Gap: adopting stage-specific system prompts wholesale would conflict with
Newmark's existing modes and stable-prefix goals. The useful follow-up is an
experiment comparing small stage metadata plus bounded tool surfaces against
full prompt replacement, not a direct port.

| --- | --- | --- | --- | --- |

| Context-aware compression | Atomic durable replacement plus request-only continuation | No unified retention benchmark | Add deterministic compression fixtures and scored continuity audit | Active task and evidence retained; completed/stale tasks not resumed; no immediate recompression loop |
| Active tool discovery | `tool_provision` compact catalog and exact schema grants | Token/success trade-off not quantified | Add a representative large-tool workload and provisioning metrics | Lower attached-schema tokens without lower completion/precision; policy boundaries unchanged |
| Parallel independent tools | Parallel batch plus all-receipts barrier | No consolidated latency/failure trajectory report | Add batch timing and receipt-completeness fixture | Wall time approaches max independent latency; every success/failure receipt reaches next provider turn |
| Interrupt and event inbox | Guide/Next queues, abort paths, runtime ownership and deferred receipts | Evidence spread across suites | Add one end-to-end response trajectory fixture | Exact interruption point, no lost Guide/Next, no duplicate turn, clean recovery and one final answer |

## Reproducible offline experiments

### Async Agent

Commands were run from `chapter4/async-agent` with `PYTHONUTF8=1`:

```text
python demo.py parallel
python demo.py interrupt
python demo.py state
```

Results:

- Four independent simulated I/O tools: serial `4.53 s`, parallel `1.52 s`,
  measured speedup `2.98x`, close to the `1.50 s` theoretical lower bound.
- Three running tasks were interrupted around `12-36%`, all produced cancelled
  state, and a new task then completed to `100%` in the same runtime.
- A checkpoint preserved three trajectory events and two task progress values;
  a fresh runtime restored both tasks as suspended at their recorded progress.

Windows compatibility finding: without `PYTHONUTF8=1`, the upstream demo reaches
its logic but fails while printing `check mark`, `stop sign` and related Unicode
characters through the GBK console. The reference source was not modified.

### Active Tool Discovery

Command run from `chapter8/active-tool-discovery`:

```text
python demo.py --offline --output results/newmark-dev-0.1.2-offline.json
```

Results for 126 tools and eight scripted tasks:

| Strategy | Precise tasks | Total injected tokens | Average tokens | Average latency |
| --- | ---: | ---: | ---: | ---: |
| Full injection | 8/8 | 93,040 | 11,630 | 0.038 s |
| One-shot prefilter | 4/8 | 8,236 | 1,030 | 0.028 s |
| Active discovery | 8/8 | 7,796 | 974 | 0.042 s |

The offline mechanism therefore reduced injection by approximately `11.9x`
while preserving `8/8` scripted coverage. One-shot prefiltering missed forex,
crypto, opinion/news and academic tasks. Because the model is a scripted mock,
these accuracy values validate the mechanism and failure shape, not production
model quality.

### Staged System Prompt

`python demo.py --list-stages` completed offline and confirmed three explicit
surfaces: requirements analyst, implementation engineer and code reviewer, each
with a separate tool set and transition tools. This is retained as an ablation
reference only; Newmark already exposes user-visible modes and should not copy
the prompt-swapping design without cache and behavior evidence.

The offline runs created only ignored cache, result, checkpoint and `__pycache__`
files inside the ignored upstream repository. Its tracked porcelain status
remains empty.

## Decision

No Agent kernel change is authorized by this study alone. The next implementation
slice should start with measurement infrastructure and deterministic ablations,
then promote only improvements that demonstrate a measurable benefit without
weakening current focus, policy, receipt, compression or finalization semantics.
