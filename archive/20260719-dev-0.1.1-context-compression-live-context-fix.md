# dev-0.1.1 Context Compression Live-Context Fix

Date: 2026-07-19

## Symptom

One active Build displayed repeated completed context-compression activities at short intervals. The first compression was valid, but later provider/tool subturns compressed the same historical prefix again.

## Root Cause

`transformContext` returned a compacted message array only for the current provider request. The native Agent loop retained its original mutable `context.messages`, so every tool continuation rebuilt the provider request from the stale pre-compression context. Persisted Newmark history was compacted, but the live Kernel context was not.

## Implementation

- Added a context-transform result contract with separate `messages` and optional `replacementMessages` fields.
- The native Agent loop atomically replaces both the active mutable context and `config.state.messages` when a durable replacement is returned.
- The detailed compression continuation prompt is appended only to the immediate provider request.
- The durable replacement contains the compressed history without that transient prompt.
- Added a regression scenario with a long stale prefix followed by a tool call and provider continuation. It proves compression executes once, the second provider call excludes the stale prefix and transient prompt, and new assistant/tool receipts append to the compacted context.
